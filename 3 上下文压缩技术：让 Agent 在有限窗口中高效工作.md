
# 第四章：上下文压缩技术：让 Agent 在有限窗口中高效工作

## 4.1 为什么需要上下文压缩？

大模型的上下文窗口（Context Window）是一块**昂贵且有限的"工作台"**。即便最新模型已支持 200K+ Token，实际工程中仍面临三重压力：

```
[上下文压力三角]
        ┌──────────┐
        │  成本膨胀  │  ← Token 数量直接决定 API 费用
        └────┬─────┘
             │
    ┌────────┴────────┐
    │                 │
┌───┴───┐       ┌────┴────┐
│延迟增长│       │注意力衰减│  ← Lost in the Middle
└───────┘       └─────────┘
```

- **成本**：GPT-4o 输入价格 $2.5/M Token，一次 100K Token 的调用就是 $0.25。长程任务可能产生数百次调用。
- **延迟**：上下文越长，首 Token 延迟（TTFT）越高。128K 上下文的 TTFT 可达数秒。
- **注意力衰减**：研究表明，模型对长文本中间区域的注意力显著下降（"Lost in the Middle" 现象）[^1]。

**核心矛盾**：Agent 需要足够的历史信息来保持连贯性，但窗口塞得越满，质量和成本越失控。上下文压缩就是在"记得住"和"塞得下"之间寻找最优平衡。

## 4.2 上下文管理的物理约束

### 4.2.1 "迷失中间"现象（Lost in the Middle）

实验表明，模型对长文本的**首尾信息敏感，中间区域注意力显著衰减**。当上下文超过 32K 时，关键指令若藏在中间，极易被忽略或曲解。

```
[注意力权重分布示意]
权重 ↑
     | ██                          ██  ← 首尾高亮
     | ██                          ██
     | ██░░░░░░░░░░░░░░░░░░░░░░░░░░██
     |_______________________________→ 序列位置
       开头                    结尾
```

> **来源**：Liu et al., "Lost in the Middle: How Language Models Use Long Contexts", 2023 [^1]

### 4.2.2 位置编码外推的代价

Transformer 依赖位置编码（如 **RoPE**）感知序列顺序。通过 NTK-Aware Scaling 等技术可将训练长度外推至更长序列，但代价是中间区域注意力进一步分散，幻觉率上升。

**工程启示**：长窗口 ≠ 长记忆。超过 64K 后，模型实际有效利用率通常不足 40%。必须用压缩策略对冲，而非单纯依赖窗口扩容。

## 4.3 主流压缩策略全景

### 4.3.1 滑动窗口 + 关键帧保留

最简单的策略：丢弃最早的历史轮次，但强制保留关键信息。

```
[滑动窗口示意]
时间 →
┌───┬───┬───┬───┬───┬───┬───┬───┐
│ S │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ 7 │  ← 原始对话
└───┴───┴───┴───┴───┴───┴───┴───┘
         ↓ 窗口滑动
┌───┬───────────┬───┬───┬───┬───┐
│ S │  [丢弃]   │ 4 │ 5 │ 6 │ 7 │  ← S=System Prompt 永久保留
└───┴───────────┴───┴───┴───┴───┘
```

- **优势**：实现简单，成本低
- **劣势**：可能丢失关键历史决策
- **适用**：多轮闲聊、客服对话等短依赖场景

### 4.3.2 滚动摘要压缩（Rolling Summarization）

当窗口触及阈值，调用模型将旧对话压缩为结构化摘要，作为"伪历史"注入新上下文。

```
[滚动压缩流程]
┌──────────────┐     ┌─────────────┐     ┌──────────────────┐
│ 轮次 1-10    │ ──→ │ 摘要模型提炼 │ ──→ │ "用户要求重构     │
│ (原始对话)   │     │             │     │  auth 模块，已确认│
└──────────────┘     └─────────────┘     │  JWT 方案"       │
                                         └────────┬─────────┘
                                                  │
┌──────────────┐                                  │
│ 轮次 11-20   │ ← 新对话 + 上述摘要 ─────────────┘
│ (继续执行)   │
└──────────────┘
```

- **优势**：保留语义连续性，Token 消耗降低 60%+
- **劣势**：摘要过程本身消耗 Token，且可能丢失细节
- **适用**：长程编码任务、多阶段工作流

### 4.3.3 工具结果截断（Tool Result Truncation）

Agent 调用工具（如读文件、执行命令）返回的结果往往极长。直接全量注入会瞬间击穿上下文预算。

```
[工具结果截断策略]
┌─────────────────────────────────────────┐
│ 原始工具输出 (50,000 chars)             │
├─────────────────────────────────────────┤
│ ████████████████████████████████████████│ ← 头部保留
│ ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░│ ← 中间省略
│ ████████████████████████████████████████│ ← 尾部保留（含错误信息）
├─────────────────────────────────────────┤
│ ⚠️ [Content truncated]                 │
└─────────────────────────────────────────┘
```

**关键设计**：尾部通常包含错误堆栈、执行结果等关键信息，截断时必须采用 **Head+Tail 策略**，而非简单截头。

### 4.3.4 预防性上下文守卫（Preemptive Context Guard）

不等溢出再压缩，而是在每次 LLM 调用前主动检查上下文总量，按"最旧优先"策略压缩已有的工具结果。

```
[预防性守卫流程]
                    ┌──────────────┐
  每次 LLM 调用前 → │ 估算总 Token  │
                    └──────┬───────┘
                           │
                    ┌──────┴───────┐
                    │ 超预算？      │
                    └──┬───────┬───┘
                   Yes │       │ No
                       ↓       ↓
              ┌────────────┐  正常调用
              │ 从最旧的工具│
              │ 结果开始压缩│
              └────────────┘
```

### 4.3.5 策略对比

| 策略              | Token 节省 | 信息保真度 | 实现复杂度 | 适用场景           |
|-------------------|-----------|-----------|-----------|-------------------|
| 滑动窗口          | ★★★★     | ★★       | ★         | 短依赖对话         |
| 滚动摘要          | ★★★      | ★★★★    | ★★★      | 长程任务           |
| 工具结果截断      | ★★★★★   | ★★★     | ★★       | 工具密集型 Agent   |
| 预防性守卫        | ★★★★    | ★★★     | ★★★     | 通用 Agent 运行时  |
| 结构化标签包裹    | ★★       | ★★★★★  | ★        | 所有场景           |

## 4.4 开源实战：OpenClaw 的三层压缩架构

OpenClaw 是一个开源 AI Agent 框架，其上下文压缩系统采用**三层递进式架构**，是工业级压缩策略的典型实现[^2]。

```
[OpenClaw 三层压缩架构]
┌─────────────────────────────────────────────────────┐
│ Layer 3: Compaction（会话级滚动摘要）                │
│ 触发：上下文溢出 or 手动触发                         │
│ 策略：分块摘要 → 合并 → 替换历史                     │
├─────────────────────────────────────────────────────┤
│ Layer 2: Context Guard（预防性工具结果压缩）          │
│ 触发：每次 LLM 调用前自动检查                        │
│ 策略：最旧工具结果优先压缩为占位符                    │
├─────────────────────────────────────────────────────┤
│ Layer 1: Tool Result Truncation（单条结果截断）       │
│ 触发：单条工具返回超过上下文窗口 30%                  │
│ 策略：Head+Tail 截断，保留错误信息                    │
└─────────────────────────────────────────────────────┘
```

### 4.4.1 Layer 1：工具结果截断

OpenClaw 对每条工具返回设置了**硬性上限**：单条结果不得超过上下文窗口的 30%，且绝对值不超过 400K 字符。

截断时采用**智能 Head+Tail 策略**：检测尾部是否包含错误/诊断信息（通过正则匹配 `error`、`exception`、`traceback` 等关键词），若有则保留尾部 30% 预算，中间插入省略标记。

```
[Head+Tail 截断逻辑]
if 尾部包含错误信息:
    head_budget = 总预算 × 70%
    tail_budget = 总预算 × 30%（最多 4000 chars）
    结果 = head + "⚠️ [middle omitted]" + tail
else:
    结果 = head + "⚠️ [truncated]"
```

> **来源**：`src/agents/pi-embedded-runner/tool-result-truncation.ts` [^2]

### 4.4.2 Layer 2：预防性上下文守卫

在每次 LLM 调用前，Context Guard 会估算当前上下文总字符数，与预算（上下文窗口 × 75%）比较。超预算时，从**最旧的工具结果**开始逐条替换为占位符 `[compacted: tool output removed to free context]`，直到总量回到预算内。

```
[Context Guard 预算计算]
上下文预算 = contextWindowTokens × 4 chars/token × 0.75 (headroom)
单条上限   = contextWindowTokens × 3.5 chars/token × 0.5

示例（128K 窗口）：
  预算 ≈ 384,000 chars
  单条上限 ≈ 224,000 chars
```

这一层的关键设计是**不触发额外 LLM 调用**——纯字符串替换，零额外成本。

> **来源**：`src/agents/pi-embedded-runner/tool-result-context-guard.ts` [^2]

### 4.4.3 Layer 3：会话级滚动摘要（Compaction）

当前两层仍无法控制上下文增长时，触发完整的 Compaction 流程：

```
[Compaction 流程]
┌──────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────┐
│ 估算历史  │ →  │ 按 Token 份额│ →  │ 逐块调用模型 │ →  │ 合并多段  │
│ Token 总量│    │ 切分为 N 块  │    │ 生成摘要     │    │ 摘要      │
└──────────┘    └──────────────┘    └──────────────┘    └──────────┘
                                                              │
                                          ┌───────────────────┘
                                          ↓
                                   ┌──────────────┐
                                   │ 摘要替换旧历史│
                                   │ 保留近期对话  │
                                   └──────────────┘
```

**核心设计要点**：

1. **自适应分块比例**：根据消息平均大小动态调整。消息越大，分块越小（`BASE_CHUNK_RATIO=0.4`，最低 `MIN_CHUNK_RATIO=0.15`），避免单块超出模型处理能力。

2. **安全边距**：Token 估算采用 `chars/4` 启发式，但多字节字符、特殊 Token 会导致低估。OpenClaw 统一乘以 1.2 倍安全系数（`SAFETY_MARGIN=1.2`）。

3. **标识符保留策略**：摘要时强制保留 UUID、文件路径、API Key 等不可重建的标识符，防止摘要后丢失关键引用。

4. **渐进式降级**：若完整摘要失败（如单条消息过大），自动跳过超大消息，仅摘要可处理部分，并附注 `[Large toolResult (~50K tokens) omitted from summary]`。

> **来源**：`src/agents/compaction.ts` [^2]

### 4.4.4 三层协作的效果

```

无压缩:    ████████████████████████████████ 溢出！(200K+ tokens)
```

## 4.5 结构化上下文工程

压缩不仅是"删减"，更是"重组"。用结构化标签包裹上下文，能显著提升模型对信息的利用效率。

```xml
<context>
  <system_rules>安全边界与角色设定</system_rules>
  <compacted_summary>前 50 轮对话的结构化摘要</compacted_summary>
  <recent_history turns="5">最近 5 轮完整对话</recent_history>
  <tool_output type="error" tool="shell">
    npm ERR! code ENOENT
    npm ERR! syscall open
  </tool_output>
</context>
```

**优势**：降低解析歧义，模型对结构化数据的注意力分配更均匀，Token 利用率提升 20%~30%。

## 4.6 生产环境避坑指南

### 4.6.1 摘要质量保障

滚动摘要的最大风险是**信息丢失**。关键缓解措施：

- **任务状态必须保留**：活跃任务、批量操作进度（如"5/17 项已完成"）、待办事项不可丢弃
- **最近意图优先**：摘要指令中明确要求"优先保留近期上下文"
- **标识符零容忍**：UUID、哈希、文件名等不可重建的标识符必须原样保留

### 4.6.2 Token 估算的陷阱

`chars / 4` 是常用的 Token 估算启发式，但存在系统性低估：
- 中文/日文等多字节字符：1 字符 ≈ 1-2 Token（而非 0.25）
- 代码中的特殊符号：`===`、`>>>` 等可能被拆分为多个 Token
- JSON 结构：大量引号和括号增加 Token 密度

**解法**：统一乘以 1.2~1.5 倍安全系数，或使用 `tiktoken` 等精确分词器做关键路径校验。

### 4.6.3 多模态 Token 膨胀

一张 1024×1024 图片经 Vision Encoder 可转化为 1000~2000 个 Token。直接全量注入会瞬间击穿预算。

- **解法**：前置裁剪（Crop）、降采样（Downscale）、关键区域提取（OCR/目标检测），仅将高信息密度区域送入 LLM。

## 本章小结

- **长窗口 ≠ 长记忆**：Lost-in-the-Middle 与外推衰减是物理规律，必须用压缩策略对冲
- **压缩是分层的**：单条截断 → 预防性守卫 → 滚动摘要，三层递进，各司其职
- **Head+Tail 截断**：工具结果的尾部往往包含最关键的错误信息，截断时必须保留
- **摘要质量是生命线**：标识符保留、任务状态保留、渐进式降级是工业级摘要的三大支柱
- **结构化标签**：用 XML/JSON 包裹上下文，提升模型注意力分配效率

> **开发者 Checklist**
> 1. [ ] 是否对工具返回结果设置了单条 Token 硬上限？是否采用 Head+Tail 截断保留错误信息？
> 2. [ ] 长任务是否引入滚动摘要？摘要指令是否要求保留任务状态与标识符？
> 3. [ ] Token 估算是否加入安全系数？多模态输入是否经过前置裁剪/降采样？
