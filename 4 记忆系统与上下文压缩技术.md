



# 第四章：记忆系统与上下文压缩技术

## 4.1 记忆机制的层级架构：Agent 的“大脑分区”
Agent 的记忆不是单一缓存，而是分层协作的系统。类比人类认知，可划分为三层：


```
[记忆分层架构示意]
┌─────────────────────────────────────────┐
│ 程序性记忆 (Procedural)                 │ ← 固化工作流、工具调用范式、安全边界
│ (微调权重 / System Prompt 模板)         │   特点：稳定、低延迟、难动态修改
├─────────────────────────────────────────┤
│ 长期记忆 (Long-term)                    │ ← 向量数据库、知识图谱、用户画像
│ (Embedding + Retrieval)                 │   特点：持久化、按需检索、支持跨会话
├─────────────────────────────────────────┤
│ 短期记忆 (Working)                      │ ← 当前对话窗口、KV Cache、工具返回
│ (Context Window)                        │   特点：高速访问、易丢失、成本敏感
└─────────────────────────────────────────┘
```

**工程启示**：不要把所有信息塞进短期记忆。程序性记忆靠 Prompt 模板固化，长期记忆靠 RAG 外挂，短期记忆仅保留**当前任务必需**的活跃状态。分层设计是控制 Token 成本与延迟的第一道防线。

## 4.2 上下文管理的核心挑战：遗忘曲线与位置编码

### 4.2.1 “迷失中间”现象（Lost in the Middle）
实验表明，模型对长文本的**首尾信息敏感，中间区域注意力显著衰减**。当上下文超过 32K 时，关键指令若藏在中间，极易被忽略或曲解。


[注意力权重分布示意]
权重 ↑
     | ██                          ██  ← 首尾高亮
     | ██                          ██
     | ██░░░░░░░░░░░░░░░░░░░░░░░░░░██
     |_______________________________→ 序列位置
       开头                    结尾


### 4.2.2 位置编码外推：如何突破 200K+ 限制？
Transformer 本身无顺序概念，依赖位置编码（Positional Encoding）。主流模型采用 **RoPE（旋转位置编码）**，通过向量旋转角度编码相对位置。
- **外推原理**：训练时只见过 4K 长度，推理时通过缩放旋转频率（NTK-Aware Scaling），让模型“ extrapolate ”到更长序列，无需重新训练。
- **代价**：外推倍数越高，中间区域注意力越分散，幻觉率上升。

**工程启示**：长窗口 ≠ 长记忆。超过 64K 后，模型实际有效利用率通常不足 40%。必须配合压缩策略，否则只是“能塞进去”，而非“真记得住”。

## 4.3 上下文压缩策略实战：从“全量塞入”到“精准投喂”

### 4.3.1 滑动窗口 + 关键帧保留
- **机制**：丢弃最早的历史轮次，但强制保留 System Prompt、用户核心目标、关键工具返回结果（如报错堆栈、API 成功响应）。
- **适用**：多轮调试、客服对话。成本低，实现简单。

### 4.3.2 分层摘要与滚动压缩（Rolling Summarization）
- **机制**：当窗口触及阈值，调用轻量模型（如 Qwen-7B / Llama-3-8B）将旧对话压缩为结构化摘要，作为“伪历史”注入新 Prompt。

[滚动压缩流程]
[轮次1-5] → 摘要模型提炼 → "用户要求重构 auth 模块，已确认 JWT 方案"
[轮次6-10] → 新对话 + 上述摘要 → 继续执行

- **适用**：长程任务、多阶段工作流。保留语义连续性，Token 消耗降低 60%+。

### 4.3.3 RAG vs 全量注入决策矩阵
| 场景                | 推荐策略          | 原因                          |
|---------------------|-------------------|-------------------------------|
| 精准事实查询        | RAG（向量+关键词）| 避免噪声干扰，按需加载        |
| 强逻辑依赖/代码生成 | 全量+AST裁剪      | 需保持上下文连贯性            |
| 超长文档/日志分析   | 混合检索+摘要     | 平衡覆盖率与成本              |

### 4.3.4 结构化上下文工程
放弃纯文本拼接，改用 XML/JSON 标签包裹：
```xml
<context>
  <system_rules>...</system_rules>
  <recent_history turns="3">...</recent_history>
  <tool_output type="error">...</tool_output>
</context>
```
**优势**：降低解析歧义，模型对结构化数据的注意力分配更均匀，Token 利用率提升 20%~30%。

## 4.4 企业级案例：Claude Code & Cursor 的上下文工程

### Cursor：基于 AST 的依赖图谱注入
Cursor 不盲目 `cat` 整个仓库。它解析代码的 **抽象语法树（AST）**，构建函数/类/模块的调用依赖图。当用户提问时，仅将**直接相关文件 + 上下游依赖**注入上下文，无关文件彻底过滤。
- **效果**：10万行代码库的上下文从 ~50K Token 压缩至 ~3K Token，响应速度提升 5 倍，幻觉率骤降。

### Claude Code / Manus：长程任务的上下文修剪
面对持续数小时的自动化任务，Manus 采用**状态对象（State Object）管理**：
- 维护 `current_goal`, `completed_steps`, `open_issues` 等结构化字段。
- 每轮迭代后，自动清理已解决的分支上下文，仅保留活跃路径与待办事项。
- 结合 Checkpoint 机制，实现断点续跑与上下文无损恢复。

## 4.5 🔍生产环境三大隐形坑（主动补充）

### 4.5.1 记忆淘汰策略：FIFO 已死，语义打分当立
简单按时间丢弃（FIFO）会误删关键信息。工业级系统采用**语义重要性评分**：
- 基于工具返回值类型（错误日志 > 成功响应 > 普通确认）。
- 结合用户显式标记（如“请记住这条规则”）。
- 低分条目优先压缩或归档至长期记忆。

### 4.5.2 跨会话记忆与隐私合规
- **用户画像持久化**：将偏好、历史决策沉淀为向量，新会话启动时自动检索注入。
- **PII 脱敏**：记忆入库前必须经过正则/NLP 管道过滤手机号、身份证、密钥。合规红线不可触碰。

### 4.5.3 多模态 Token 膨胀效应
一张 1024x1024 图片经 Vision Encoder 可转化为 1000~2000 个 Token。直接全量注入会瞬间击穿上下文预算。
- **解法**：前置裁剪（Crop）、降采样（Downscale）、关键区域提取（OCR/目标检测），仅将高信息密度区域送入 LLM。

## 本章小结
- **记忆需分层**：短期管执行，长期管知识，程序性管规范。混用必崩。
- **长窗口≠记忆**：Lost-in-the-Middle 与外推衰减是物理规律，必须用压缩策略对冲。
- **压缩是门艺术**：滑动窗口保底，滚动摘要续命，RAG 精准检索，结构化标签提效。
- **企业实践验证**：Cursor 的 AST 裁剪与 Manus 的状态修剪证明：上下文管理的核心是“相关性过滤”，而非“容量堆砌”。

> 3. [ ] 多模态输入是否经过前置裁剪/降采样？是否设置单会话多模态 Token 硬上限？
