# 第五章：工程落地与最佳实践：从 Demo 到 Production 的跨越

## 5.1 结构化 Prompt 与防御性设计：把提示词当代码写
提示词不是自然语言草稿，而是**模型与系统间的 API 契约**。自由文本极易导致指令漂移、格式崩溃与安全漏洞。

### 5.1.1 结构化模板：划定语义边界
使用 XML/Markdown 标签显式分隔不同语义块，强制模型按区块解析：
```xml
<system_instruction>
  你是资深后端工程师。仅输出 JSON，勿包含解释性文字。
</system_instruction>
<context>
  <user_query>优化以下 SQL 查询...</user_query>
  <schema>CREATE TABLE orders (...)</schema>
</context>
<output_format>
  {"optimized_sql": "...", "reasoning": "..."}
</output_format>
```
**工程启示**：结构化边界可使解析失败率下降 70%+。模型对标签内内容的注意力分配更均匀，显著降低“指令淹没”风险。

### 5.1.2 防御性设计：防注入、防幻觉、防崩溃
- **输入清洗**：过滤恶意 Payload（如 `</system_prompt>` 闭合攻击）、限制最大长度。
- **输出校验**：使用 Pydantic/Zod 严格校验 JSON Schema。失败则触发自动重试（最多 2 次），重试失败走降级逻辑。
- **置信度拦截**：要求模型输出 `confidence_score`。低于阈值（如 0.7）时自动转人工或切换备用策略。

## 5.2 评估体系（Evaluation）：从“感觉不错”到“数据说话”
生产环境不能依赖主观体验。必须建立自动化评估流水线，量化模型表现。

```
[离线评估流水线]
Golden Dataset (覆盖边界/异常/多轮) 
        ↓
   批量推理 (Batch Inference)
        ↓
  指标计算 (RAGAS / DeepEval / 规则脚本)
        ↓
  可视化看板 (准确率/忠实度/延迟/成本)
```

**核心指标**：
- **忠实度（Faithfulness）**：回答是否严格基于检索上下文？（防幻觉）
- **上下文精度（Context Precision）**：检索到的片段是否真正相关？（防噪声）
- **答案相关性（Answer Relevance）**：是否直接解决用户问题？（防跑题）

**工程启示**：每次 Prompt 迭代或模型升级前，必须跑通回归测试。评估数据集需持续从线上 Bad Case 中沉淀，形成“数据飞轮”。

---

## 5.3 线上监控与可观测性（Observability）
日志不能只记“成功/失败”。Agent 系统需全链路追踪，实现分钟级故障定位。

- **Trace 追踪**：为每次请求生成唯一 `trace_id`，串联 Prompt 组装、模型调用、工具执行、结果返回的全生命周期。
- **核心仪表盘**：
  - `TTFT / TPS`：性能健康度
  - `Token Cost / Request`：成本水位
  - `Tool Success Rate`：执行稳定性
  - `Fallback Trigger Count`：降级触发频率
- **工具推荐**：LangSmith、Arize Phoenix、OpenTelemetry + Prometheus/Grafana。

**工程启示**：可观测性是生产环境的“神经系统”。无监控不上线，否则故障排查将退化为“盲人摸象”。

---

## 5.4 成本与性能权衡：模型路由与缓存策略

### 5.4.1 智能路由（Model Routing）
并非所有请求都需要 GPT-4/Claude-3.5。通过轻量级 Router（或小模型）按任务复杂度分发：
```
[路由决策树]
用户请求 → 意图分类器
   ├─ 简单格式化/分类 → 路由至 7B/8B 模型 (低成本, 低延迟)
   ├─ 代码生成/复杂推理 → 路由至 70B+ 模型 (高成本, 高精度)
  └ 敏感/合规查询 → 路由至 本地微调模型 + 规则引擎
```
**工程启示**：合理路由可削减 40%~60% 的 Token 账单，同时保障核心场景的体验。

### 5.4.2 Prompt Caching（前缀缓存）
系统提示词、RAG 检索结果、长代码库往往高度重复。开启 Prompt Caching 后，服务端会复用已计算的 KV Cache，后续请求只需计算增量部分。
- **效果**：重复上下文的首字延迟（TTFT）降低 80%，输入 Token 成本下降 50%。
- **注意**：缓存命中率依赖 Prompt 结构的稳定性。频繁变动的前缀会失效。

## 5.5 🔍生产环境三大隐形坑（主动补充）

### 5.5.1 Prompt 版本控制与 CI/CD
Prompt 即代码（Prompt-as-Code）。必须纳入 Git 管理，配套 Staging 环境验证、A/B 测试与灰度发布。禁止直接在生产环境手改 Prompt。

### 5.5.2 降级与熔断策略（Circuit Breaker）
LLM 服务存在抖动风险。必须设计多级降级链：
`LLM 正常响应 → 超时/报错 → 切换备用小模型 → 命中规则引擎/缓存 → 返回友好提示/转人工`
配合熔断器（如 Resilience4j），连续失败阈值触发后自动短路，保护下游系统雪崩。

### 5.5.3 数据隐私与成本归因
- **PII 脱敏**：日志落盘与缓存前，必须通过正则/NLP 管道过滤手机号、身份证、密钥。合规红线不可妥协。
- **成本分摊**：为每个请求打上 `tenant_id`、`feature_tag` 标签。实现按部门/业务线核算 ROI，避免“糊涂账”。

## 本章小结
- **提示词工程化**：结构化模板与防御性校验是稳定性的基石。
- **评估驱动迭代**：建立 Golden Dataset 与自动化指标看板，用数据替代直觉。
- **全链路可观测**：Trace 追踪与核心仪表盘是快速排障与成本优化的前提。
- **架构防御优先**：模型路由、前缀缓存、降级熔断、Prompt 版本控制与隐私脱敏，共同构成生产级 Agent 系统的护城河。
