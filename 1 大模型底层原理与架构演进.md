# 第一章：大模型底层原理与架构演进

## 1.1 架构演进：从 Encoder-Decoder 到 Decoder-only 的必然选择

### 1.1.1 RNN 的瓶颈与 Attention 的破局
在 Transformer 问世前，序列建模高度依赖 RNN/LSTM。其核心缺陷在于**串行依赖**与**长程信息衰减**。每个时间步的隐藏状态必须等待上一步计算完成，导致 GPU 并行算力无法释放；同时，梯度在反向传播时随序列长度指数级衰减，模型难以捕捉跨段落的语义关联。

```
[RNN 串行计算示意]
t=1: 输入"我" → 计算 h1 → 输出预测
t=2: 输入"爱" → 依赖 h1 计算 h2 → 输出预测  (必须等待)
t=3: 输入"学" → 依赖 h2 计算 h3 → 输出预测
...
t=n: 信息经过 n 次传递，早期信号已被严重稀释
```

Transformer 通过 **Self-Attention** 彻底重构了信息交互方式：序列中任意两个 Token 可直接计算关联权重，路径长度恒为 1。这不仅消除了递归依赖，还使模型具备全局感受野，为后续架构分化奠定基础。

### 1.1.2 原始 Transformer：Encoder 与 Decoder 的分工与协同
原始 Transformer 专为机器翻译设计，采用经典的 **Encoder-Decoder** 架构。两模块职责明确且通过注意力机制紧密耦合：

```
[原始 Transformer 数据流示意]
源句子: "I love AI."  →  [Encoder]  →  上下文矩阵 C (双向编码)
                                      ↓ (Cross-Attention)
目标句子: "<SOS>"     →  [Decoder]  →  生成 "我" → "爱" → "AI" → "<EOS>"
                      (Masked Self-Att + Cross-Att + FFN)
```

- **Encoder 的作用与机制**：
  - **双向上下文建模**：接收完整源序列，通过多层 Self-Attention 让每个 Token 同时关注前后文，生成富含全局语义的隐藏表示。
  - **适用场景**：适合需要“深度理解”的任务，如文本分类、命名实体识别。典型代表为 BERT 等 Encoder-only 模型。
- **Decoder 的作用与机制**：
  - **自回归生成**：按从左到右顺序逐个生成目标 Token。
  - **双重注意力设计**：
    1. **Masked Self-Attention**：通过下三角掩码屏蔽未来 Token，保证因果性。
    2. **Cross-Attention**：Query 来自 Decoder，Key/Value 来自 Encoder 输出，实现源语言与目标语言的语义对齐。

### 1.1.3 为什么工业界全面转向 Decoder-only？
尽管 Encoder-Decoder 结构严谨，但当前主流大模型（GPT、Claude、Llama）均转向 **Decoder-only** 架构。这一转变由训练范式、扩展性与工程效率共同驱动：

```
[Decoder-only 因果掩码示意]
输入序列: [我] [爱] [学] [习]
注意力权重矩阵 (上三角被 Mask 遮挡为 -∞
      我   爱   学   习
我   [✓]  [✗  [✗  [✗  ← 只能看自己
爱   [✓]  [✓]  [✗  [✗  ← 能看"我"和自己
学   [✓]  [✓]  [✓]  [✗  ← 能看"我","爱"和自己
习   [✓]  [✓]  [✓]  [✓]  ← 能看所有历史
```

**核心优势**：
1. **任务统一性**：几乎所有 NLP 任务均可转化为文本补全形式。问答、翻译、代码生成只需调整 Prompt，无需维护独立的 Encoder 权重或 Cross-Attention 模块。
2. **数据扩展性**：Encoder-Decoder 依赖高质量平行语料，获取成本高；Decoder-only 仅需海量无标注文本进行 Next Token Prediction，可无损利用互联网全量数据。
3. **推理与部署简化**：生成阶段仅需维护单一权重栈与 KV Cache，显存占用更紧凑。省去 Cross-Attention 的同步开销，首字延迟（TTFT）与吞吐（TPS）更易优化。
4. **隐式推理涌现**：因果语言建模迫使模型在预测下一个 Token 时，内部构建完整的逻辑链。这种自回归压力比双向掩码更能激发复杂推理能力。

## 1.2 训练范式：预训练与后训练（Post-Training）的协同

### 1.2.1 预训练（Pre-training）：构建通用知识基座
预训练是模型的“基座构建期”。模型在万亿级 Token 语料上执行 **Next Token Prediction**，通过最小化交叉熵参数。

```
[预训练目标示意]
输入 Prompt: "今天天气真___"
模型输出概率分布:
  好: ████████████████████ 80%
  坏: ████ 10%
  冷: ██ 5%
  差: █ 3%
  其他: █ 2%
目标: 调整参数，使正确词的概率最大化
```

**工程关键**：预训练决定“智商下限”。数据清洗质量（去重、去毒、代码/数学配比）直接决定能力天花板。算力投入需遵循 Compute-Optimal 原则，避免参数冗余或数据不足。

### 1.2.2 后训练（Post-Training）总览：从“通才”到“专才”的桥梁
现代大模型研发中，**后训练（Post-Training）** 已成为独立且关键的研发阶段。它泛指预训练完成后，为使模型具备指令遵循、安全可控、领域适配等能力而进行的一系列微调与对齐操作。

```
[后训练管线示意]
Base Model (预训练基座)
       ↓
[阶段1] SFT (指令微调) → 学会听懂指令、按格式输出
       ↓
[阶段2] Preference Alignment (偏好对齐) → RLHF / DPO / ORPO
       ↓
[阶段3] Safety & Domain Tuning (安全/领域强化) → 红蓝对抗、垂直行业数据注入
       ↓
Instruct / Chat Model (最终可用版本)
```

**为什么需要后训练？**
- Base Model 本质是“概率续写引擎”，直接提问常得到冗长、跑题或危险内容。
- 后训练通过分布偏移（Distribution Shift），将模型从“模仿互联网文本”引导至“遵循人类交互规范”。

### 1.2.3 指令微调（SFT）：格式化输出与任务适配
SFT 使用高质量 `(Instruction, Response)` 对进行有监督微调，是后训练的第一步。

```
[SFT 数据格式示例]
User: 请将以下英文翻译成中文："Hello, world!"
Assistant: 你好，世界！
---
User: 写一段 Python 代码计算斐波那契数列第 10 项。
Assistant: def fib(n):
    if n <= 1: return n
    return fib(n-1) + fib(n-2)
```

**机制与工程实践**：
- 冻结大部分底层表征，微调顶层注意力与前馈网络，使输出分布向“结构清晰、语气可控”迁移。
- 数据多样性（多任务、多格式、多语言）比单纯的数据量更重要。通常 1万~10万 条精心构造的指令数据即可显著提升可用性。

### 1.2.4 偏好对齐（RLHF/DPO）：价值观校准与安全边界
SFT 后的模型仍可能生成有害、偏见或过度自信的内容。对齐阶段引入人类价值观约束：

```
[RLHF vs DPO 流程对比]
RLHF (三步法):
  生成样本 → 人工排序 → 训练奖励模型(RM) → PPO优化策略(最大化奖励+KL约束)
  
DPO (直接优化):
  收集偏好对 (优选回答 y_w, 劣选回答 y_l) → 直接更新策略网络
  数学等价于 RLHF，但跳过 RM 与 PPO，训练更稳定、显存占用更低
```

**工程意义**：对齐决定了模型的“安全性”与“实用性”。企业级应用必须在此阶段注入领域规范、合规边界与拒答策略（Refusal Patterns）。

## 1.3 能力涌现：规模定律与推理机制的质变

### 1.3.1 Scaling Laws 与 Compute-Optimal 训练
模型性能 $L$ 与参数量 $N$、训练数据量 $D$ 呈平滑的幂律关系：$L \propto N^{-\alpha} + D^{-\beta}$。

```
[Scaling Laws 趋势图 (概念)]
性能 ↑
     |        /
     |      /
     |    /
     |  /
     |/___________→ 参数量/数据量/算力
```

**工程启示**：
- 能力增长是可预测的，不存在绝对的“魔法阈值”。
- **Chinchilla 定律**指出：算力固定时，应均衡扩大模型规模与训练数据量。盲目堆参数而数据不足会导致过拟合与算力浪费。

### 1.3.2 思维链（CoT）：隐式推理的显式化
当模型跨越 ~10B 参数临界点后，会自发涌现逐步推理能力。CoT 并非额外训练模块，而是**提示工程对隐式能力的解锁**。

```
[CoT 效果对比]
标准 Prompt: "小明有 5 个苹果，吃了 2 个，又买了 3 个，现在有几个？"
模型输出: 6 (错误，直接猜)

CoT Prompt: "... Let's think step by step."
模型输出:
  1. 初始: 5 个
  2. 吃了 2 个: 5 - 2 = 3 个
  3. 买了 3 个: 3 + 3 = 6 个
  答案: 6 个 (正确，路径清晰)
```

**本质与工程价值**：
- 模型在预训练阶段已吸收大量人类解题步骤。CoT 引导模型将内部隐式计算路径显式输出为中间 Token。
- 中间步骤可作为调试依据，便于定位幻觉来源或逻辑断裂点，显著提升复杂任务（数学、代码、规划）的准确率。

## 本章小结
- **架构演进**：原始 Transformer 通过 Encoder（双向理解）与 Decoder（自回归生成+交叉对齐）协同工作。工业界转向 Decoder-only 是因其在任务统一性、数据扩展性、推理效率与隐式推理涌现上的综合优势。
- **训练流水线**：预训练构建知识基座 → 后训练（SFT + 对齐 + 安全/领域强化）实现指令跟随与价值观校准。后训练是现代大模型从“可用”到“好用”的核心壁垒。
- **能力边界**：Scaling Laws 提供可预测的增长曲线，CoT 等提示技术可激活模型已有的隐式推理潜力。

> **延伸阅读与实操建议**：
