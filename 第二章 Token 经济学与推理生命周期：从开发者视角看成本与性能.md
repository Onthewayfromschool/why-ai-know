# 第二章：Token 经济学与推理生命周期：从开发者视角看成本与性能

## 2.1 为什么 Token 消耗越来越大？Agent 时代的“上下文通胀”
早期聊天机器人是“一问一答”，Token 消耗极低。但现代 AI Coding 与 Agent 工具（如 Cursor、Claude Code、GitHub Copilot）的工作模式已彻底改变：
**模型不再是“对话者”，而是“带环境感知的执行节点”**
。

```
[典型 AI Coding 会话的 Token 构成]
System Prompt: 角色设定、代码规范、安全边界      (~500 tokens)
Conversation History: 多轮调试记录、报错回溯      (~2,000 tokens)
Context Injection: 当前文件、依赖文件、终端输出   (~15,000 tokens)
Tool Results: 文件读取结果、API 返回、AST 解析    (~3,000 tokens)
---------------------------------------------------------
Total Input:  ~20,500 tokens  →  用户实际看到的“回答”可能只有 200 tokens
```

**为什么消耗暴涨？**
- **环境透视需求**：Agent 需要“看到”文件系统、依赖树、运行时日志才能精准操作。这些上下文必须完整注入 Prompt。
- **多轮交互累积**：调试是迭代过程。每一轮的错误堆栈、模型反思、修正代码都会追加到历史窗口中。
- **工具链开销**：每次调用外部工具（读文件、执行命令、查文档）的返回结果都会作为新 Token 喂回模型。

。**工程启示**：Token 消耗已从“交互成本”转变为“行动成本”。开发者必须意识到：**长上下文不是免费的午餐，而是换取精准执行的必要投入**。盲目塞入全量代码库会导致成本失控与响应延迟，需通过智能裁剪与检索替代。

## 2.2 输入 vs 输出：不对称的账单与性能瓶颈
在实际使用中，输入（Input/Prompt）与输出（Output/Completion）的 Token 消耗呈现明显的不对称性。理解这一点是控制成本与优化体验的关键。

### 2.2.1 成本结构：谁在真正“烧钱”？
- **输入 Token（读）**：通常单价较低（约为输出的 1/3~1/5），但**体积庞大**。在 Agent 场景中，输入占比常达 80%~95%。
- **输出 Token（写）**：单价较高，但**体积较小**。模型倾向于精简回答，除非被要求详细解释或生成完整文件。

```
[成本与体积对比示意 (AI Coding 场景)]
          体积占比        单价权重        实际成本占比
输入:     ████████████ 90%  ▓▓░░░░░░░ 低     ██████████ 70%
输出:     ██░░░░░░░░░░ 10%  ▓▓▓▓▓▓▓▓▓▓高     ███░░░░░░░ 30%
```

### 2.2.2 性能瓶颈：延迟到底卡在哪？
- **输入主导首字延迟（TTFT）**：模型必须“读完”所有输入 Token 才能开始生成。输入越长，等待第一个字出现的时间越久。
- **输出主导生成速度（TPS）**：一旦开始输出，速度取决于模型逐个吐出 Token 的能力。输出越长，总耗时越久。

**开发者视角**：当你觉得“AI 反应慢”，通常是输入上下文太长导致 Prefill 阶段卡顿；当你觉得“AI 写得慢”，通常是输出阶段受限于内存带宽。两者优化手段完全不同。

## 2.3 推理生命周期：Prefill（读）与 Decode（写）的算力真相
大模型推理并非匀速流水线，而是由两个物理特征截然不同的阶段拼接而成。这与输入/输出的不对称性直接对应。

### 2.3.1 Prefill 阶段：并行“阅卷”
- **对应输入**：一次性处理所有 Prompt Token。
- **计算特征**：**计算密集型（Compute-bound）**。所有 Token 的注意力矩阵可并行计算，GPU 算力满载。
- **体验指标**：**TTFT（Time To First Token）**。决定用户按下回车后多久能看到第一个字。

### 2.3.2 Decode 阶段：串行“答题”
- **对应输出**：自回归逐个生成新 Token。每生成一个，就追加到上下文，作为下一步的输入。
- **计算特征**：**内存带宽密集型（Memory-bound）**。每次只算 1 个 Token，但需读取全部模型权重 + 历史 KV Cache。GPU 算力闲置，瓶颈卡在显存读取速度。
- **体验指标**：**TPS（Tokens Per Second）**。决定长代码块或详细解释的生成流畅度。

```
[Prefill vs Decode 资源消耗对比]
阶段      | 对应环节   | 瓶颈资源      | 优化手段                 | 用户感知
----------|------------|---------------|--------------------------|----------
Prefill   | 处理输入   | GPU 算力      | 增大 Batch、编译优化、剪枝 | 首字快慢
Decode    | 生成输出   | 显存带宽      | 量化、KV 优化、流式输出    | 打字速度
```

**工程启示**：
- **AI Coding 场景**：输入通常极长（文件+历史），Prefill 耗时占比高。优化重点应是**上下文压缩**与**前缀缓存**。
- **内容生成场景**：输出较长，Decode 耗时占比高。优化重点应是**量化加速**与**批量调度**。

## 2.4 KV Cache 与显存：长上下文的“隐形税”
为什么模型能“记住”之前的对话？因为推理引擎在后台默默维护着一张巨大的缓存表：**KV Cache**。

### 2.4.1 缓存机制：用空间换时间
在 Decode 阶段，如果每次生成新 Token 都重新计算前面所有内容的 Attention，计算量将呈平方级爆炸。KV Cache 的核心逻辑是：**历史 Token 的 Key 和 Value 不会变，直接缓存起来，新 Token 只需计算自己的 Q/K/V 并与缓存做匹配**。

```
[KV Cache 增长示意 (单会话)]
Step 0: 输入 10,000 Token → 缓存占用 10,000 单位
Step 1: 生成 100 Token    → 缓存占用 10,100 单位
Step 2: 生成 100 Token    → 缓存占用 10,200 单位
...
上下文越长，显存线性增长，直至触发 OOM（显存溢出）
```

### 2.4.2 显存占用公式与硬性约束
KV Cache 是推理显存的最大消耗者，其占用量可精确估算：
$$ \text{VRAM}_{\text{KV}} = 2 \times \text{Layers} \times \text{HiddenSize} \times \text{SeqLen} \times \text{BatchSize} \times \text{BytesPerParam} $$
*(系数 2 代表 K 和 V 各一份；SeqLen = 输入 + 输出总长度)*

**工程启示**：
- **上下文窗口 ≠ 免费额度**：支持 128K 上下文意味着需要数倍于 4K 的显存。企业需根据业务场景合理设置 `max_tokens`。
- **碎片化难题**：传统框架为每个请求分配连续显存块，但不同请求长度不一，会产生大量碎片，导致显存利用率仅 60%~70%。
- **PagedAttention 破局**：vLLM 等现代框架借用操作系统“虚拟内存分页”思想，将 KV Cache 切分为固定物理块，逻辑连续、物理分散，显存利用率可提升至 95% 以上，并支持多请求共享同一份前缀缓存（如重复的系统提示词或代码库索引）。

## 本章小结
- **上下文通胀是 Agent 的必然代价**：环境感知、多轮调试、工具返回导致输入 Token 体积庞大，Token 已从交互成本转为行动成本。
- **输入与输出高度不对称**：输入主导成本与首字延迟（Prefill），输出主导生成速度与单价（Decode）。优化策略需按场景分化。
- **KV Cache 是显存黑洞**：长上下文直接线性消耗显存。生产环境必须依赖 PagedAttention、前缀缓存与量化技术，否则并发能力将严重受限。

> **实操建议（开发者视角）**：
> 1. **上下文瘦身**：在 AI Coding 中，优先使用 AST 依赖分析或语义检索（RAG）注入相关文件，而非全量 `cat *.py`。
> 2. **监控双指标**：在业务日志中同时记录 `input_tokens`、`output_tokens` 与 `ttft/tps`。输入占比过高时检查 Prompt 冗余；输出 TPS 过低时检查显存带宽或启用量化。
> 3. **框架选型**：部署 Agent 服务时，禁用原生 `transformers.generate()`，改用 vLLM 或 SGLang，并开启 `enable_prefix_caching=True`，可显著降低重复上下文的 Prefill 开销。
