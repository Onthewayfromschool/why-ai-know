
# 第五章：记忆系统：让 Agent 拥有持久化的知识与经验

## 5.1 为什么 Agent 需要记忆？

上一章讨论了如何在**单次会话**中高效管理上下文。但现实中，Agent 面临的挑战远不止于此：

```
[无记忆 Agent 的困境]
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 会话 A    │    │ 会话 B    │    │ 会话 C    │
│ "我喜欢   │    │ "我喜欢   │    │ "我喜欢   │
│  暗色主题" │    │  什么主题？│    │  什么主题？│  ← 每次都要重新告知
│           │    │  不知道…  │    │  不知道…  │
└──────────┘    └──────────┘    └──────────┘
     ↓               ↓               ↓
  会话结束          会话结束          会话结束
  记忆清零          记忆清零          记忆清零
```

没有记忆系统的 Agent 就像一个**每天失忆的同事**——每次对话都从零开始，无法积累经验、无法记住用户偏好、无法延续上次的工作进度。

**记忆系统的核心价值**：
- **跨会话连续性**：记住用户偏好、历史决策、项目规范
- **知识积累**：将过往经验沉淀为可检索的知识库
- **个性化服务**：基于历史交互提供定制化响应

---

## 5.2 记忆的层级架构：Agent 的"大脑分区"

类比人类认知，Agent 的记忆可划分为三层：

```
[记忆分层架构]
┌─────────────────────────────────────────┐
│ 程序性记忆 (Procedural)                 │ ← 固化工作流、安全边界
│ (System Prompt / 微调权重)              │   特点：稳定、低延迟、难动态修改
├─────────────────────────────────────────┤
│ 长期记忆 (Long-term)                    │ ← 向量数据库、知识图谱、用户画像
│ (Embedding + Retrieval)                 │   特点：持久化、按需检索、跨会话
├─────────────────────────────────────────┤
│ 短期记忆 (Working)                      │ ← 当前对话窗口、KV Cache
│ (Context Window)                        │   特点：高速访问、易丢失、成本敏感
└─────────────────────────────────────────┘
```

| 记忆类型   | 载体                    | 生命周期     | 访问速度 | 典型用途             |
|-----------|------------------------|-------------|---------|---------------------|
| 程序性记忆 | System Prompt / 权重    | 永久        | 极快    | 角色设定、安全规则    |
| 长期记忆   | 向量数据库 / 文件系统    | 跨会话持久  | 中等    | 用户偏好、项目知识    |
| 短期记忆   | Context Window / Cache  | 单次会话    | 极快    | 当前任务状态         |

**工程启示**：不要把所有信息塞进短期记忆。程序性记忆靠 Prompt 模板固化，长期记忆靠 RAG 外挂，短期记忆仅保留**当前任务必需**的活跃状态。

## 5.3 长期记忆的存储与索引

### 5.3.1 存储形式：从纯文本到结构化索引

最直觉的记忆存储方式是**纯文本文件**（如 Markdown），人类可读、易于编辑。但纯文本无法高效检索，因此需要构建索引层。

```
[存储与索引双层架构]
┌──────────────────────────────────┐
│ 原始存储层 (Source of Truth)      │
│ ┌──────────┐  ┌────────────────┐ │
│ │MEMORY.md │  │ memory/*.md    │ │  ← 人类可读、可编辑
│ └──────────┘  └────────────────┘ │
└──────────────┬───────────────────┘
               │ 文件变更监听
               ↓
┌──────────────────────────────────┐
│ 索引层 (SQLite + Embedding)      │
│ ┌──────┐ ┌────────┐ ┌─────────┐ │
│ │chunks│ │chunks_  │ │chunks_  │ │  ← 分块 + 向量化 + 全文索引
│ │      │ │vec     │ │fts      │ │
│ └──────┘ └────────┘ └─────────┘ │
└──────────────────────────────────┘
```

这种"**原始文件 + 索引数据库**"的双层设计兼顾了可维护性与检索效率。

### 5.3.2 内容分块（Chunking）

长文档不能整篇存入向量数据库，需要切分为语义完整的片段。常见策略：

```
[Markdown 分块示意]
原始文档（500 行）
┌────────────────────────────┐
│ ## 项目架构                 │ ─┐
│ 本项目采用微服务架构...      │  │ Chunk 1 (≤512 tokens)
│ 主要模块包括...             │ ─┘
│ ## 数据库设计                │ ─┐
│ 使用 PostgreSQL...          │  │ Chunk 2 (≤512 tokens)
│ 表结构如下...               │ ─┘
│ ## API 设计                 │ ─┐
│ RESTful 风格...             │  │ Chunk 3 (≤512 tokens)
│ ...                        │ ─┘
└────────────────────────────┘
         ↕ overlap（重叠区域，保持语义连贯）
```

**关键参数**：
- **chunk_size**：每块最大 Token 数（通常 256~1024）
- **overlap**：相邻块的重叠 Token 数（通常 chunk_size 的 10%~20%），防止语义断裂

### 5.3.3 向量化（Embedding）

将文本块转化为高维向量，使语义相近的内容在向量空间中距离更近。

```
[Embedding 流程]
"本项目采用 JWT 认证"  ──→  [0.12, -0.34, 0.56, ..., 0.78]  (768/1536 维)
"用户登录使用 Token"   ──→  [0.11, -0.32, 0.55, ..., 0.77]  ← 语义相近，向量相似
"数据库使用 PostgreSQL" ──→  [-0.45, 0.67, -0.12, ..., 0.23] ← 语义不同，向量远离
```

**主流 Embedding 模型**：

| 模型                    | 维度  | 特点                    |
|------------------------|-------|------------------------|
| OpenAI text-embedding-3 | 1536  | 质量高，需 API 调用      |
| Gemini Embedding        | 768   | Google 生态集成          |
| BGE / GTE（本地）       | 1024  | 可离线运行，隐私友好     |
| Voyage AI               | 1024  | 代码语义优化             |

## 5.4 混合检索：向量 + 关键词的最佳组合

纯向量检索擅长语义匹配，但对精确关键词（如函数名、错误码）表现不佳。工业级系统采用**混合检索**策略。

### 5.4.1 双路召回

```
[混合检索架构]
                    用户查询
                       │
              ┌────────┴────────┐
              ↓                 ↓
     ┌────────────────┐ ┌──────────────┐
     │ 向量检索        │ │ BM25 关键词   │
     │ (语义相似度)    │ │ (精确匹配)    │
     │ "认证方案"      │ │ "JWT"        │
     │  → 相关段落     │ │  → 精确命中   │
     └────────┬───────┘ └──────┬───────┘
              │                │
              └────────┬───────┘
                       ↓
              ┌────────────────┐
              │ 分数融合 + 排序 │  ← 加权合并、去重、MMR 多样性
              └────────┬───────┘
                       ↓
                  Top-K 结果
```

### 5.4.2 BM25 全文搜索

BM25 是经典的信息检索算法，基于词频（TF）和逆文档频率（IDF）计算相关性。对于精确关键词匹配（如 `NullPointerException`、`user_id`），BM25 远优于向量检索。

现代实现通常基于 **SQLite FTS5** 扩展，无需额外部署搜索引擎：

```sql
-- 创建全文索引表
CREATE VIRTUAL TABLE chunks_fts USING fts5(
  text,
  id UNINDEXED,
  path UNINDEXED,
  source UNINDEXED
);

-- 查询示例
SELECT * FROM chunks_fts WHERE chunks_fts MATCH 'JWT AND authentication';
```

### 5.4.3 分数融合策略

两路召回的分数量纲不同，需要归一化后加权合并：

```
最终分数 = α × 向量相似度(归一化) + (1-α) × BM25分数(归一化)
```

- **α 通常取 0.5~0.7**：偏重语义匹配
- **MMR（最大边际相关性）**：在排序时惩罚与已选结果过于相似的候选，提升结果多样性
- **时间衰减**：可选地对较旧的记忆降权，优先返回近期内容

## 5.5 开源实战：OpenClaw 的记忆系统

OpenClaw 的记忆系统是"**Markdown 文件 + SQLite 索引 + 混合检索**"架构的典型实现[^1]。

### 5.5.1 存储架构

```
[OpenClaw 记忆存储结构]
~/.openclaw/agents/<agentId>/
├── MEMORY.md              ← 主记忆文件（用户/Agent 可直接编辑）
├── memory/
│   ├── project-rules.md   ← 项目规范
│   ├── decisions.md       ← 历史决策记录
│   └── preferences.md     ← 用户偏好
└── .memory-index.db       ← SQLite 索引数据库
```

**数据源类型**：
- **memory**：`MEMORY.md` 和 `memory/` 目录下的 Markdown 文件
- **sessions**：历史会话记录（可选开启），将过往对话也纳入可检索范围

### 5.5.2 索引数据库 Schema

OpenClaw 使用 Node.js 内置的 `node:sqlite` 模块，零外部依赖：

```
[SQLite 表结构]
┌─────────────────────────────────────────────────┐
│ files        │ 文件元信息（路径、哈希、修改时间）  │
├──────────────┼──────────────────────────────────┤
│ chunks       │ 文本块（路径、行号、哈希、向量）    │
├──────────────┼──────────────────────────────────┤
│ chunks_vec   │ 向量索引（sqlite-vec 扩展）        │
├──────────────┼──────────────────────────────────┤
│ chunks_fts   │ 全文索引（FTS5 扩展）              │
├──────────────┼──────────────────────────────────┤
│ embedding_   │ Embedding 缓存（避免重复调用 API） │
│ cache        │                                   │
└──────────────┴──────────────────────────────────┘
```

**Embedding 缓存**是一个精巧的设计：对于未变更的文本块，直接复用缓存的向量，避免重复调用 Embedding API，显著降低成本。缓存键为 `(provider, model, provider_key, text_hash)` 的四元组。

> **来源**：`src/memory/memory-schema.ts` [^1]

### 5.5.3 写入流程：从文件变更到索引更新

```
[索引更新流程]
┌──────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ 文件变更  │ →  │ chokidar     │ →  │ 扫描 .md 文件│ →  │ 按 Token 分块│
│ (保存/新建)│    │ 文件监听     │    │ 计算文件哈希 │    │ (带 overlap) │
└──────────┘    └──────────────┘    └──────────────┘    └──────┬───────┘
                                                              │
                    ┌─────────────────────────────────────────┘
                    ↓
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ 查缓存       │ →  │ 未命中则调用 │ →  │ 写入 chunks  │
│ (text_hash)  │    │ Embedding API│    │ + vec + fts  │
└──────────────┘    └──────────────┘    └──────────────┘
```

**分块参数**（默认值）：
- `chunk_size`：512 tokens（即 ~2048 字符）
- `overlap`：chunk_size 的 10%（~50 tokens）

分块逻辑会在换行符边界切割，避免在句子中间断开。超长单行会被强制按 `maxChars` 切分。

> **来源**：`src/memory/internal.ts` — `chunkMarkdown()` 函数 [^1]

### 5.5.4 检索流程：Agent 如何使用记忆

OpenClaw 为 Agent 提供两个记忆工具：

**`memory_search`**：语义搜索，返回最相关的文本片段

```
[memory_search 调用示例]
Agent 收到问题："这个项目的认证方案是什么？"
  │
  ↓ 自动调用 memory_search
  │
  query: "项目认证方案"
  │
  ↓ 混合检索（向量 + BM25）
  │
  返回: [
    { path: "memory/decisions.md", lines: "12-18",
      snippet: "## 认证方案\n采用 JWT + Refresh Token..." },
    { path: "MEMORY.md", lines: "45-50",
      snippet: "安全要求：所有 API 必须经过 JWT 验证..." }
  ]
```

**`memory_get`**：按路径和行号读取原文片段（用于在 `memory_search` 找到线索后获取完整上下文）

```
Agent: memory_get(path="memory/decisions.md", from=10, lines=20)
  │
  ↓ 返回第 10-30 行的原始 Markdown 内容
```

这种"**先搜后取**"的两步设计，避免了一次性注入大量记忆内容，保持上下文精简。

> **来源**：`src/agents/tools/memory-tool.ts` [^1]

### 5.5.5 多 Embedding Provider 支持与降级

OpenClaw 支持多种 Embedding 提供商，并实现了自动降级：

```
[Provider 降级链]
首选 Provider（如 OpenAI）
  │
  ├─ 成功 → 使用
  │
  └─ 失败（配额耗尽/网络错误）
       │
       ↓
  Fallback Provider（如本地模型）
       │
       ├─ 成功 → 使用（记录降级原因）
       │
       └─ 失败 → 记忆搜索不可用（返回 disabled=true）
```

当 Provider 不可用时，`memory_search` 不会抛异常，而是返回结构化的错误信息，Agent 可据此向用户解释情况。

## 5.6 记忆淘汰与隐私合规

### 5.6.1 语义重要性评分

简单按时间丢弃（FIFO）会误删关键信息。工业级系统采用**语义重要性评分**：

- 基于内容类型：错误日志 > 决策记录 > 普通确认
- 结合用户显式标记：如"请记住这条规则"
- 访问频率加权：频繁被检索命中的记忆提升权重

### 5.6.2 跨会话记忆与 PII 脱敏

- **用户画像持久化**：将偏好、历史决策沉淀为向量，新会话启动时自动检索注入
- **PII 脱敏**：记忆入库前必须经过正则/NLP 管道过滤手机号、身份证、密钥等敏感信息。合规红线不可触碰

```
[PII 脱敏管道]
原始文本: "联系人张三，手机 13812345678，密钥 sk-abc123..."
    │
    ↓ 正则 + NER 过滤
    │
脱敏文本: "联系人张三，手机 [PHONE]，密钥 [SECRET]..."
    │
    ↓ 写入记忆存储
```

## 5.7 RAG vs 全量注入：何时用记忆检索？

| 场景                | 推荐策略          | 原因                          |
|---------------------|-------------------|-------------------------------|
| 精准事实查询        | RAG（向量+关键词）| 避免噪声干扰，按需加载        |
| 强逻辑依赖/代码生成 | 全量+AST 裁剪     | 需保持上下文连贯性            |
| 超长文档/日志分析   | 混合检索+摘要     | 平衡覆盖率与成本              |
| 用户偏好/历史决策   | 记忆系统自动注入  | 跨会话持久化，无需用户重复说明 |

**工程启示**：记忆系统和 RAG 不是互斥的。记忆系统是 Agent 的"个人笔记本"，RAG 是"公司知识库"。前者存储个性化经验，后者存储通用知识。两者协同才能构建真正智能的 Agent。

## 本章小结

- **记忆需分层**：短期管执行，长期管知识，程序性管规范。混用必崩
- **存储与索引分离**：原始 Markdown 保证可读性，SQLite 索引保证检索效率
- **混合检索是标配**：向量检索捕捉语义，BM25 捕捉精确关键词，融合后效果远超单一方式
- **先搜后取**：`memory_search` + `memory_get` 两步走，避免一次性注入过多记忆
- **Embedding 缓存**：对未变更内容复用缓存向量，是控制 API 成本的关键优化
- **隐私合规是底线**：PII 脱敏管道必须前置于记忆写入

> **开发者 Checklist**
> 1. [ ] 是否实现了记忆的分层架构？长期记忆是否独立于上下文窗口持久化存储？
> 2. [ ] 检索是否采用混合策略（向量 + 关键词）？是否实现了 Embedding 缓存避免重复 API 调用？
> 3. [ ] 记忆写入前是否经过 PII 脱敏？跨会话记忆是否符合数据合规要求？
